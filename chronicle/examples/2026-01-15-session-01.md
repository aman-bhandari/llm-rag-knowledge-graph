# Session 1: The Wrapper That Lost Its Name

*January 15, 2026. The first session in the log. A decorator that worked, and a test that proved the learner did not understand why.*

---

## The Opening

She came in with a decorator that worked.

That was the first problem. A decorator that works but is not understood is worse than a decorator that throws a clear error, because nothing is pulling the learner toward the mechanism. The function ran. The output looked right. The test passed. Everything said "done."

The session was supposed to be introductory — decorators as a pattern, written one way, explained once, moved on. It turned into an hour-long detour because of a single assertion in a test she had not yet written but was about to.

The question on the table when the session started: *what does `@log_execution` actually do to the function underneath it?*

She answered: "It wraps it." That was the opening analogy. She pictured the original function as a gift and the decorator as wrapping paper around it. The gift stays the same; the paper is added on top.

The analogy was close enough to be useful. It was also wrong in a specific way that would cost an hour to find.

---

## The Work

She wrote the decorator first.

```python
def log_execution(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        result = func(*args, **kwargs)
        print(f"{func.__name__} returned {result}")
        return result
    return wrapper

@log_execution
def add(a, b):
    return a + b
```

Called `add(2, 3)`. Output: `Calling add`, `5`, `add returned 5`. Test green. One assertion: `assert add(2, 3) == 5`. Passed immediately.

Then she wrote the second test. A sanity check, she thought. The kind of test you write to prove the thing you think is true.

```python
def test_decorator_preserves_name():
    assert add.__name__ == "add"
```

That one failed.

`add.__name__` returned `"wrapper"`.

---

## The Break

She stared at it.

The failure message was clear — `AssertionError: assert 'wrapper' == 'add'` — and completely disorienting. The decorator had "wrapped" the function. The function was still called `add` when she invoked it. Why was its name `wrapper`?

For the next fifteen minutes she tried the wrong debugging strategy: rereading the decorator definition, looking for a typo, suspecting Python version behavior. She changed the inner function's name to `add_wrapper` thinking maybe that would help. It did not — now `add.__name__` was `add_wrapper`.

The coach let her stay in the break.

Eventually she asked a question that was more useful than any of her attempts to fix the code: "Why is the outer function's name showing up at all? Isn't `add` separate from the wrapper?"

That was the crack in the analogy.

She had pictured the decorator as adding paper around a gift. But the decorator did not add paper around `add`. It replaced `add` with `wrapper`. The name `add` in the module namespace no longer pointed at the function she had written — it pointed at the function the decorator had returned. The original `add` was still alive, but only inside `wrapper`'s closure, not in the namespace.

When she saw that, she went quiet for about thirty seconds, and then said — almost to herself, not to the coach — "Oh. The outer function is not a wrapper around the inner. It replaces it. The inner function is only reachable through the closure."

---

## The Click

The fix was `functools.wraps`:

```python
import functools

def log_execution(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        result = func(*args, **kwargs)
        print(f"{func.__name__} returned {result}")
        return result
    return wrapper
```

Four characters of import, one line of decoration on the inner function. `add.__name__` now returned `"add"`. The second test passed.

But the four characters were not the lesson. The lesson was that she could not have arrived at `functools.wraps` from inside the "wrapping a gift" analogy. The analogy pushed her toward "what am I wrapping wrong" instead of "what does replacement mean". `functools.wraps` is a patch on the mechanism. You only need it once you have understood that the decorator is doing replacement, and that the replacement loses the original's metadata unless you copy it explicitly.

She wrote the cross-reference to closures in her own words, in a wiki node we will link below. The closure is the part of the mechanism that lets `wrapper` still find `func` after `add` has been rebound. Without the closure, the decorator is impossible; with the analogy of wrapping, the closure is invisible. Shifting the analogy from "wrapping" to "replacing" made the closure load-bearing in her mental model for the first time.

---

## Closing

She shipped the fix at minute 62 of a session that was budgeted for 20.

That is a fair trade. A twenty-minute session that ends with `@log_execution` working and no understanding of why costs the learner a week later, when the decorator silently breaks some other library's introspection and the debugging has to start from scratch. A sixty-minute session that ends with "the outer function replaces the inner, the closure is how the inner stays reachable, `functools.wraps` copies metadata from the replaced onto the replacement" costs the learner nothing — that model is portable to every decorator she will ever write.

The concepts touched: [[decorators]], [[iterator-protocol]] (referenced via the closure explanation — closures are how iterators also keep state across `__next__` calls). Both have their own wiki nodes.

Next session on the schedule: generators. She is already asking whether generators and iterators are the same thing. That is the hook for session 2.
